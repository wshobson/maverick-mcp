# Phase 8: Server Assembly and Cutover Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Build `maverick/server/` assembling every domain's tools on FastMCP, close the tool-parity ledger with explicit port-or-retire rulings, delete the legacy `maverick_mcp/` package entirely, and ship v1.0.0 as `maverick-mcp-server`.

**Architecture:** Phase 8 of the modernization. The 2026-07-20 server recon (`.superpowers/sdd/p8-recon-server.md`, gitignored scratch; load-bearing facts restated here) is the map. The legacy surface is 119 tools / 9 prompts / 11 resources; the new curated surface lands at ~39 existing + the ports below. Assembly wiring: one shared `Engine` + `Cache` → `MarketDataService` first (eager schema; `build_mover_fetcher` must not be forgotten) → screening/portfolio/technical inject it → backtesting/research construct only behind their extras → per-domain `configure(service)` then `register(mcp)`.

**Tech Stack:** Python 3.12+ (the `<3.13` pin dies with ta-lib), FastMCP, maverick.platform.

## Global Constraints

Gates: eight-tree pytest (`tests/portfolio tests/screening tests/technical tests/market_data tests/platform tests/structure tests/backtesting tests/research`) plus the new `tests/server` tree, `make lint`, tracked-files `ruff format --check` plus explicit format-check of changed files, `uv run ty check maverick` (the legacy trees leave the strict zone as they die), `make docs-check` where docs change. Files under 500 lines. Commit per task; stage explicitly. Base install boots with core tools only; extras degrade to zero registrations with one warning each (established pattern). After cutover, no `maverick_mcp` reference may survive anywhere (code, config, CI, docs, packaging).

## Parity ledger rulings (the 43-gap adjudication)

Every legacy tool without a prior decision now has one. PORT: risk dashboard (Phase 4's logged intent), watchlist, trade journal — genuine personal-trader features consumed by the owner. RETIRE (rationale logged per group below): signal engine, health monitoring, introspection, decision-log/registry-status, news/Adanos sentiment, cache introspection, economic calendar, legacy prompts (superseded by 3 new curated prompts), legacy resources, the `get_watchlist` demo tool.

- 2026-07-20 (addendum, Task 6 review): A SECOND console-script entry `maverick-mcp-server = "maverick.server.app:main"` is added alongside `maverick-mcp` (both point at the same `main`). Reason: the registry `server.json` declares `identifier: maverick-mcp-server` with `runtime_hint: uvx`, so a registry client running the natural `uvx maverick-mcp-server` needs a console script matching the package name to resolve; without the alias only `uvx --from maverick-mcp-server maverick-mcp` works, which server.json has no field to express. The alias makes the bare form resolve directly with zero risk (wheel scoping/entrypoint unaffected). The shorter `maverick-mcp` name stays for humans.

- 2026-07-20 (addendum, Task 1 review): Risk dashboard pricing intentionally diverges from legacy's per-position path: legacy's router fetched a 7-day OHLC lookback via `StockDataProvider.get_stock_data` for each position; the ported service reuses the domain's existing `_fetch_quote_prices` (live quote, Semaphore-guarded) for consistency with `get_portfolio`'s own pricing — both answer "current price," and re-deriving a second untested lookback path would have reintroduced the exact legacy-vs-domain-layer divergence Phase 4 flagged. Not a regression.
- 2026-07-20 (addendum, Task 2 review): `portfolio_watchlist_remove` returns an honest `removed: bool` reflecting whether a row was actually deleted, diverging from the legacy router's unconditional `{"removed": True}` — deliberate, matches this package's own `data.py::delete_position` convention. Disclosed, not a fidelity bug.
- 2026-07-20 (addendum, Task 3 review): `strategy_performance` intentionally carries no `created_at`/`updated_at` columns, unlike `journal_entries` — legacy's `StrategyPerformance` model extends only `Base`, not `TimestampMixin`, and adding the columns would break inserts against a pre-existing legacy-shaped table lacking them. Pinned by a dedicated test so the asymmetry can't regress.
- 2026-07-20 (addendum, Task 3 review): `profit_factor` faithfully preserves legacy's actual (uncapped) behavior — `float("inf")` when a strategy has zero losing trades — even though legacy's own docstring comment claims it is "capped for serialization"; the real code path was ported, not the stale comment. `r_multiple` is likewise carried forward as a schema field but is always `None` in practice: legacy declares it but no service/router/analytics code path ever sets it, so it's ported as a documented dead field rather than dropped or invented.
- 2026-07-20 (addendum, Task 3 review): `JournalService` could not be composed inside `PortfolioService` (already at the 500-line file-size cap with zero headroom) — it ships as a fully standalone top-level domain service (own engine, own lazily-created schema, own session factory) and is wired into `portfolio.tools.configure(service, journal_service=...)` via an optional parameter. Task 4's server assembly constructs it alongside, not through, `PortfolioService`.
- 2026-07-20 (addendum, Task 6 review): Dockerfile `HEALTHCHECK` removed outright rather than switched to a `maverick-mcp --help` probe: the new server exposes no HTTP `/health` endpoint (it's an MCP server, not a REST API), and a `--help` probe would always pass without observing whether the running process is actually alive. Also, the Docker image installs both `[backtesting]` and `[research]` extras by default (full tool surface, ~200MB heavier) rather than base-only, since `docker-compose.yml` targets a complete self-hosted deployment; reversible with one flag, judgment call not a hard requirement.
- 2026-07-20 (addendum, Task 7 review): `docs/exec-plans/tech-debt-tracker.md` swept during the docs pass — removed rows whose debt was structurally resolved by Task 5's legacy-package deletion. The tracker's own header rule requires removing a row "in the same change that removes the debt"; the debt (code under `maverick_mcp/`) was removed by Task 5, but nobody had swept the tracker since, so Task 7 closed that gap.

```
maverick.server -> {every maverick.<domain>.tools + config} -> ...
```
