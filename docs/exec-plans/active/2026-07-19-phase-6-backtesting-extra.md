# Phase 6: Backtesting Extra Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Port the live backtesting surface into `maverick/backtesting/` as an optional capability behind a `[backtesting]` dependency extra, delete the confirmed-dead backtesting code, and keep the server booting cleanly with zero backtesting tools when the extra is absent.

**Architecture:** Phase 6 of the modernization, following the established domain template with pure-compute data-tier siblings. The 2026-07-19 backtesting recon (`.superpowers/sdd/p6-recon-backtesting.md`, gitignored scratch; facts restated here) is the map. Behavioral references: `maverick_mcp/backtesting/**` and `maverick_mcp/api/routers/backtesting.py`; characterization fixtures follow the deterministic-seed patterns already used in `tests/test_ml_strategies.py` and `tests/integration/test_full_backtest_workflow.py`.

**Tech Stack:** Python 3.12, pydantic, pandas/numpy in core; vectorbt, numba, pandas-ta, scikit-learn, scipy only inside the extra. maverick.platform, maverick.market_data, pytest with TDD.

## Global Constraints

Same gates as Phase 5: six-tree pytest (`tests/portfolio tests/screening tests/technical tests/market_data tests/platform tests/structure`) plus the new `tests/backtesting` tree, `make lint`, tracked-files `ruff format --check`, `uv run ty check maverick maverick_mcp/services maverick_mcp/domain`, `make docs-check` where docs change. Files under 500 lines. Commit per task; stage explicitly. No `maverick_mcp` imports from `maverick/` (contract-enforced). Every module that imports an extra-only package must be import-guarded so the base install never raises at import time; `tests/backtesting` gates on `pytest.importorskip("vectorbt")` (or the narrower package a test needs) so the base-install suite stays green.

## Layer contract (Task 0 encodes)

```
maverick.backtesting: tools -> service -> {store, engine, optimization, analysis, strategies} -> config -> types
```

The `strategies/` subpackage (base, templates, ml modules) is one data-tier sibling. Only `service`/`tools` may import `maverick.market_data`; `store` may import `maverick.platform.db`; engine/optimization/analysis/strategies import neither market_data nor other domains (extend the import-linter contracts accordingly, mirroring how `maverick.technical` narrowed its independence contract). `maverick.screening`/`maverick.portfolio` stay forbidden for all backtesting modules.

### Task 2: types and config

`types.py`: pydantic models for the tool payload surface — backtest result (metrics, trade list rows, equity-curve series as `list[float]`), optimization result (grid, best params, per-combination metrics), walk-forward result, monte-carlo summary (percentile bands), strategy-comparison rows, strategy catalog entry (name, description, parameters), ML training/regime/ensemble results. Derive field sets from what the 11 legacy tools actually return today (`api/routers/backtesting.py`) — port the real shapes, don't invent. `config.py`: `BacktestingSettings` with the legacy defaults (initial capital 10000.0, fees 0.001, slippage 0.001 — confirm exact literals from `vectorbt_engine.py` and the router; env-backed `BACKTESTING_INITIAL_CAPITAL`, `BACKTESTING_FEES`, `BACKTESTING_SLIPPAGE`), plus `analysis_timeout_seconds: float = 120.0` and any grid-size/chunk limits the engine honors, using the platform env helpers and the established singleton/reset pattern. TDD round-trips and defaults/override tests mirroring prior domains. Both files must import cleanly WITHOUT the extra installed (pure pydantic). Full gate. Commit `feat(backtesting): add payload types and settings`.

### Task 3: store

`store.py`: port the five SQLAlchemy models from `maverick_mcp/data/models.py` (same table names, `extend_existing` not needed — they bind to the new domain metadata) and the CRUD operations from `maverick_mcp/backtesting/persistence.py` (save backtest result + trades, save optimization/walk-forward results, save/list backtest portfolios, fetch by id/ticker/strategy) onto `maverick.platform.db` (engine factory, ensure_schema, session scope), mirroring how `maverick/portfolio/` does persistence. Pure stdlib/pydantic/SQLAlchemy — no extra-only imports. TDD against in-memory SQLite via the platform factories: round-trip each table, FK integrity of trades to results, degraded-input validation. Full gate. Commit `feat(backtesting): add result store on the platform db seam`.

### Task 4: engine and analysis

`engine.py`: port the core of `vectorbt_engine.py` minus data fetch, caching, and charts — `run_backtest(frame, strategy_config, settings) -> ...` producing portfolio stats via vectorbt from an OHLCV DataFrame, including the memory-chunked execution path. `analysis.py`: port `maverick_mcp/backtesting/analysis.py`'s metrics/result analysis and the monte-carlo simulation logic. Both import vectorbt at module top level — that is allowed because nothing below `service` imports them in the base install path (service/tools guard). Characterization tests (`tests/backtesting/test_engine.py`, `test_analysis.py`, `pytest.importorskip("vectorbt")`): a deterministic seeded OHLCV fixture (fixed RNG seed, no network — reuse the synthetic-series pattern from `tests/test_ml_strategies.py`), run `sma_cross` end to end and pin the key metrics (total return, trade count, sharpe) to recorded values at reasonable tolerance; monte-carlo with a fixed seed pins percentile bands. Edge tests: empty frame, too-short frame. Full gate. Commit `feat(backtesting): add pure engine core and analysis`.

### Task 6: ML strategies

### Task 7: service and tools

`service.py`: `BacktestingService(market_data, settings=None)` — async methods backing all 11 tools; fetches daily history via `market_data.get_price_history` with calendar padding for warmup, runs engine/optimization/analysis/strategies, persists via `store` where the legacy tools persisted, applies `asyncio.wait_for` with `analysis_timeout_seconds`, merges the batch path (the legacy `strategy_executor`/`batch_processing` duplication collapses here: `compare_strategies` and portfolio-level runs iterate the single-strategy path with bounded concurrency). Insufficient history or empty fetch -> clear ValueError before any engine call. `tools.py`: `configure(service)`/`register(mcp)` with the 11 `backtesting_`-prefixed tools (annotations: readOnlyHint True for pure-read tools like `backtesting_list_strategies`; tools that persist results are not readOnly), consistent `{"status": "error", "error": ...}` payloads, unconfigured path, and — the phase's availability contract — `register()` detects a missing extra (guarded import) and registers nothing while logging one clear warning; the base-install server must boot with zero backtesting tools and no traceback, proven by a test that runs WITHOUT importorskip. TDD with stub market-data (deterministic frames, short-history stub, slow stub for the timeout), in-memory fastmcp Client round-trips for `backtesting_list_strategies` and `backtesting_run_backtest`, annotation assertions. Two commits allowed: `feat(backtesting): add domain service` then `feat(backtesting): add MCP tool layer with extra-guarded registration`.
