# Runbook: `Descriptors cannot be created directly` on server startup

## Symptom

`make dev`, `make dev-sse`, or `make dev-stdio` dies during import with:

```
File ".../opentelemetry/proto/common/v1/common_pb2.py", line 36, in <module>
    _descriptor.FieldDescriptor(
File ".../google/protobuf/descriptor.py", line 675, in __new__
    _message.Message._CheckCalledFromGeneratedFile()
TypeError: Descriptors cannot be created directly.
If this call came from a _pb2.py file, your generated code is out of date
and must be regenerated with protoc >= 3.19.0.
```

The backend never binds to port 8003 and `scripts/dev.sh` reports `Backend process died!`.

## Root cause

The import chain is:

```
maverick_mcp.api.server
  └── api/routers/__init__.py
      └── agents.py
          └── maverick_mcp.agents.deep_research
              └── maverick_mcp.data.vector_store (imports chromadb at module top)
                  └── chromadb
                      └── chromadb.telemetry.opentelemetry (imported unconditionally —
                          setting `ANONYMIZED_TELEMETRY=False` does NOT skip it)
                          └── opentelemetry.exporter.otlp.proto.grpc
                              └── opentelemetry.proto.common.v1.common_pb2  ← fails here
```

`chromadb 1.5.x` has a soft floor on the OTLP exporter (`opentelemetry-exporter-otlp-proto-grpc>=1.2.0`) but no upper bound, and no coupling to `opentelemetry-api` / `opentelemetry-sdk`. Under some backtracking paths, `uv`'s resolver picks **old** exporter versions (seen: `1.11.1`, released 2022) alongside a **modern** `opentelemetry-api` / `opentelemetry-sdk` (e.g. `1.41.0`). The 1.11.1 wheel ships `_pb2.py` files generated by `protoc < 3.19` that use the legacy descriptor API. The modern `protobuf` runtime (4.x, 5.x, 6.x, 7.x) rejects that API at import time.

Why does the resolver prefer the old version? When a newer `protobuf` (e.g. `7.x`) satisfies another transitive requirement, bumping the exporter to a matched otel version would force `protobuf` down to `6.x`. uv can choose to keep `protobuf` at the higher version and pin the exporter at the older one — technically a valid solution to the inequality system, but it produces this crash because the old `_pb2.py` files are not runtime-compatible with newer `protobuf`.

## Fix (now in `pyproject.toml`)

The `vectors` extra carries explicit lower bounds on the OTLP packages so the resolver is forced to match modern `opentelemetry-api` / `-sdk`:

```toml
vectors = [
    "chromadb>=0.5.0",
    "opentelemetry-exporter-otlp-proto-grpc>=1.30.0",
    "opentelemetry-proto>=1.30.0",
]
```

With these constraints:

- `opentelemetry-exporter-otlp-proto-grpc` resolves to the matched api/sdk version (e.g. `1.41.0`).
- `opentelemetry-proto` bumps to `1.41.0` (regenerated `_pb2.py` files).
- `opentelemetry-exporter-otlp-proto-common` (new package in modern otel) gets added.
- `protobuf` is auto-downgraded from `7.x` back to the compatible `6.x` line.
- `backoff` (only depended on by ancient otel exporter) is dropped.

## How to verify after a dep bump

1. Regenerate the lockfile: `uv lock`.
2. Sync with extras: `uv sync --all-extras`.
3. Confirm the matched otel set:

   ```bash
   uv run --no-sync python -c "
   from importlib.metadata import version
   for p in ['opentelemetry-api', 'opentelemetry-sdk',
            'opentelemetry-exporter-otlp-proto-grpc', 'opentelemetry-proto',
            'protobuf']:
       print(f'{p}=={version(p)}')
   "
   ```

   `api`, `sdk`, `exporter-otlp-proto-grpc`, and `proto` should all share the same `1.X.Y`.

4. Import smoke test (this is exactly what the crash prevented):

   ```bash
   uv run --no-sync python -c "
   from chromadb.auth.token_authn import TokenTransportHeader
   from opentelemetry.proto.common.v1 import common_pb2
   from maverick_mcp.data.vector_store import get_vector_store
   print('OK')
   "
   ```

5. Boot the server end-to-end:

   ```bash
   make dev-sse   # should bind to 127.0.0.1:8003 within ~20s
   ```

## Emergency workaround (do NOT commit)

If you hit this mid-debug and need to unblock *right now*:

```bash
PROTOCOL_BUFFERS_PYTHON_IMPLEMENTATION=python uv run python -m maverick_mcp.api.server --transport sse
```

This switches `protobuf` to the pure-Python parser, which accepts the legacy `_pb2.py` files. It is significantly slower and should only be used for a one-off debug session. Fix the lockfile instead.

## Do not

- Downgrade `protobuf` to `3.20.x` — many transitive deps now require `>=4`, and several require `>=5`. You will trade one crash for another.
- Set `ANONYMIZED_TELEMETRY=False` and expect it to skip the import — chromadb imports `chromadb.telemetry.opentelemetry` unconditionally; the config is only checked when telemetry events would be emitted.
- Remove `chromadb` to "fix" the import — it is a required dependency of the research vector store (`maverick_mcp/data/vector_store.py`), which the deep-research agent needs.

## CI gate — making this a blocking check

`.github/workflows/dep-smoke.yml` runs the lockfile assertion and the
import smoke test on every PR that touches `pyproject.toml`, `uv.lock`,
the workflow itself, or `scripts/check_otel_versions.py`. The workflow
failing is necessary but **not sufficient** to block a merge: GitHub
will still allow merging unless **"dep-smoke / smoke" is marked as a
required status check** in the branch protection rule for `main`.

To finish wiring this as a real gate:

1. GitHub → Settings → Branches → branch protection rule for `main`.
2. Under **Require status checks to pass before merging**, add
   `dep-smoke / smoke` to the required list.
3. Verify: open a PR that deliberately breaks the OTEL version pin
   (e.g. bump one OTEL package by hand without the matching peers) and
   confirm the "Merge" button is disabled while the workflow is red.

Until step 2 is done, a dependency-sloppy PR can still reach `main`.
